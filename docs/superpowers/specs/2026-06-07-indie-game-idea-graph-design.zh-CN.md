# 独立游戏创意图谱设计 Spec

## 1. 目的

本文档描述一个面向独立游戏创意发现系统的模块化产品设计。系统帮助独立游戏开发者从模糊兴趣和有限资源出发，逐步得到一组可解释、可比较、可进入原型验证的游戏概念。

本文档不是实现计划。它不指定数据库产品、框架、模型供应商、界面架构或部署方式。它只定义系统故事、系统边界、模块职责、数据契约和独立验收标准。

## 2. 产品命题

独立开发者通常并不需要更多随机点子。他们需要的是符合自身约束、连接真实游戏设计模式、与参考作品有明确差异，并且能够通过小型原型验证的点子。

系统应当像一位理解游戏设计的制作顾问：

1. 它把精选游戏作为设计样本进行研究。
2. 它从这些游戏中提取可复用的设计知识。
3. 它理解开发者的现实约束和兴趣偏好。
4. 它找到适合该开发者的机会区域。
5. 它形成有边界的机会框架。
6. 它将机会框架转化为具体概念卡。
7. 它指出每个概念在原型阶段必须证明什么。

系统不得承诺生成的概念一定好玩或商业成功。它的价值在于在正式开发前减少不确定性。

## 3. 范围

### 范围内

- 精选种子游戏，按设计相关性选择，而不是按数量规模选择。
- 每个种子游戏的结构化游戏设计档案。
- AI 辅助的设计标注，并带有证据、置信度和质量状态。
- 由游戏、机制、玩家体验、制作约束、创新模式和可复用参考模式构成的设计知识图谱。
- 包含约束、偏好和兴趣的开发者画像。
- 解释某个设计区域为什么适合某位开发者的机会框架。
- 只能从有边界的机会框架生成的概念卡。
- 围绕安全性、适配度、新颖度和风险的概念比较。
- 明确具体可测试假设的原型验证简报。

### 第一版范围外

- 根据用户反馈自动更新图谱的反馈闭环。
- Steam 级别的全量游戏索引。
- 声称系统能够预测游戏是否好玩或是否市场成功。
- 为每条标注设置逐条强制审批流程。
- 在没有证据、置信度和质量状态的情况下，将标注无边界地纳入图谱。
- 长期制作规划、任务估算、人力安排、预算或发行计划。

## 4. 核心流程

```text
精选游戏入库
-> 游戏设计标注
-> 设计知识图谱
-> 开发者画像
-> 机会匹配
-> 机会框架
-> 概念生成
-> 概念评估
-> 原型验证简报
```

最重要的边界位于“机会框架”和“概念生成”之间。系统不应该让语言模型基于宽泛提示自由发明。系统应先基于已知游戏设计证据组装一个受约束的框架，再只使用生成能力把这个框架具象化、表达清楚。

## 5. 系统产物

这些产物是模块之间稳定交接的对象。

### 5.1 游戏候选

一个被选择用于可能入库的游戏。

必需字段：

- 标题
- 来源引用
- 简短描述
- 选择理由

游戏候选只记录导入事实和入库意图，不要求用户在导入时判断该游戏的设计参考价值。

### 5.2 游戏设计档案

对单个游戏的结构化设计解读。

必需字段：

- 一句话设计总结
- 核心循环
- 玩家主要行动
- 玩家主要决策
- 主要玩家体验
- 主要机制
- 成长模型
- 失败模型
- 重玩性来源
- 内容结构
- 制作约束
- 创新模式
- 可复用参考模式
- 参考价值标签
- 不可复制风险
- 证据说明
- 置信度
- 质量状态

参考价值标签由游戏设计标注阶段提取，可包括：

- 机制独特
- 适合 solo 开发者参考
- 低制作成本参考
- 高系统深度参考
- 强创新模式
- 代表性类型样本

### 5.3 设计论断

关于某个游戏或设计模式的一条有证据支撑的陈述。

示例：

- “Balatro 使用玩家熟悉的扑克规则来降低学习成本。”
- “Into the Breach 通过小棋盘和完全信息制造高战术深度。”
- “抽象卡牌 UI 可以降低 solo 开发者的动画负担。”

必需字段：

- 主体
- 关系
- 客体
- 解释
- 证据
- 置信度
- 质量状态

### 5.4 开发者画像

对开发者现实条件和偏好的结构化描述。

必需字段：

- 团队规模
- 开发时间预算
- 程序能力
- 美术能力
- 音频能力
- 内容生产能力
- 偏好的游戏参考
- 讨厌的游戏参考或机制
- 期望的玩家体验
- 硬性约束
- 强偏好
- 软偏好

约束类型：

- 硬性约束：不得违反。
- 强偏好：通常应尊重，除非某个概念有异常高的收益。
- 软偏好：可以为了新颖度或适配度进行权衡。

### 5.5 机会框架

在概念生成前组装出的有边界创意简报。

必需字段：

- 目标开发者画像摘要
- 匹配到的机会区域
- 来源游戏
- 相关机制
- 相关玩家体验
- 相关制作约束
- 相关创新模式
- 推荐变形方式
- 禁止方向
- 证据路径
- 适配理由
- 风险理由

机会框架是系统防止无约束创意生成的主要机制。

### 5.6 概念卡

由一个机会框架推导出的具体游戏构思。

必需字段：

- 标题
- 一句话概念
- 核心幻想
- 核心循环
- 玩家主要决策
- 主要机制
- 参考来源
- 与参考作品的差异
- 为什么适合该开发者
- 制作风险
- 设计风险
- 新颖度理由
- 建议原型范围

### 5.7 概念评估

对概念卡进行结构化比较。

必需字段：

- 适配度评分
- 可行性评分
- 新颖度评分
- 风险评分
- 证据质量评分
- 分类

分类：

- 稳妥型：可行性高，新颖度较低。
- 平衡型：有明确新意，风险可控。
- 挑战型：新颖度更强，但会明显挑战开发者能力边界。

### 5.8 原型验证简报

对第一版原型必须测试什么的聚焦说明。

必需字段：

- 最大风险假设
- 最小原型内容
- 目标试玩时长
- 成功信号
- 失败信号
- 暂时不要做什么

简报必须避免“验证游戏是否好玩”这类泛泛表述。它应该明确可观察的玩家行为或设计属性。

## 6. 模块

每个模块都必须能通过自身输入和输出产物独立测试。任何模块都不应要求跑完整端到端流程，才能验证其基本行为。

### 6.1 精选游戏入库模块

目的：

选择并登记种子库候选游戏。

输入：

- 游戏标题或商店链接
- 可选策展备注
- 可选导入备注

输出：

- 游戏候选

职责：

- 记录该游戏为什么应进入种子库。
- 判断来源是否可用、是否重复、是否具备后续标注条件。
- 保留用户或策展者的导入备注，供后续标注阶段参考。

不得做：

- 决定最终设计标注。
- 推断完整机制或创新模式。
- 要求用户在导入时填写参考价值标签。
- 在完成设计标注前将游戏纳入设计图谱。

独立测试用例：

- 给定一款机制独特的独立游戏，模块应创建带有清晰选择理由的游戏候选。
- 给定来源信息不足、无法识别或重复导入的游戏，模块应将其标为低优先级或拒绝。
- 只给定商店链接时，模块仍应生成最小候选记录，并明确标出缺失字段。

验收标准：

- 每个被接受的候选都有入库理由。
- 每个候选都能追溯到至少一个来源引用。
- 此阶段没有任何候选被视为设计知识。

### 6.2 游戏设计标注模块

目的：

把一个游戏作为设计对象，生成结构化解读。

输入：

- 游戏候选
- 来源材料
- 可选策展备注

输出：

- 游戏设计档案
- 设计论断

职责：

- 识别核心循环、玩家行动、决策、体验、机制、创新模式和制作约束。
- 提取参考价值标签，并为标签关联证据、置信度和质量状态。
- 区分可观察事实和解释性判断。
- 为重要论断附上证据和置信度。
- 为论断分配质量状态，使低置信度材料在原型阶段仍可被谨慎使用。
- 暴露不确定性，而不是隐藏不确定性。

不得做：

- 将低置信度输出标为高置信度。
- 在没有证据的情况下把模糊论断转成图谱事实。
- 覆盖人工整理过的修正。

独立测试用例：

- 给定一款来源材料充分的已知游戏，模块应生成完整档案。
- 给定一款低美术成本但高系统深度的游戏，模块应提取对应参考价值标签，并说明证据。
- 给定来源材料较弱的游戏，模块应将置信度标低，并指出缺失证据。
- 给定互相冲突的来源材料，模块应保留不确定性，而不是静默选择一边。

验收标准：

- 每个档案都包含核心循环、机制、体验、约束、创新说明和参考价值标签。
- 每个非显而易见的论断都包含证据或明确的不确定性标记。
- 每条论断都有置信度和质量状态。

### 6.3 设计知识图谱模块

目的：

把设计知识维护为相互连接的概念。

输入：

- 游戏设计档案
- 设计论断

输出：

- 可查询的设计关系
- 证据路径
- 候选机会子图

职责：

- 将游戏连接到机制、体验、约束、风格、类型、题材、创新模式和可复用参考模式。
- 在关系上保留证据、置信度和质量状态。
- 支持从开发者约束遍历到相关游戏和设计模式。
- 支持相似游戏与对照游戏之间的比较。
- 保留低置信度关系，但必须明显降级展示。

不得做：

- 在没有标注输入的情况下发明新论断。
- 生成最终概念卡。
- 把流行度等同于设计相关性。
- 隐藏低置信度关系。

独立测试用例：

- 给定某游戏的设计论断，模块应暴露预期关系。
- 给定“低美术成本”等目标约束，模块应返回相关游戏、机制和模式。
- 给定某款游戏，模块应返回有证据支撑的路径，说明其为什么匹配某个设计模式。

验收标准：

- 每条被下游使用的关系都能追溯到有证据支撑的论断。
- 模块能返回直接匹配和多跳证据路径。
- 模块能区分强证据和弱证据。

### 6.4 开发者画像模块

目的：

以系统可使用的形式表示用户的能力、约束和兴趣。

输入：

- 用户回答
- 喜欢的游戏
- 讨厌的游戏或机制
- 期望项目规模

输出：

- 开发者画像

职责：

- 将自由文本用户输入转化为结构化约束和偏好。
- 区分硬性约束、强偏好和软偏好。
- 识别缺失的画像信息。
- 保留用户的原始表达和意图。

不得做：

- 推荐游戏或概念。
- 把所有偏好都当作硬过滤条件。
- 在未询问或未标记不确定性的情况下，静默推断关键约束。

独立测试用例：

- 给定一位美术能力弱的 solo 开发者，模块应根据用户措辞，将低美术能力记录为强偏好或硬性约束。
- 给定模糊的时间预算，模块应将画像标为不完整。
- 给定喜欢的游戏，模块应提取兴趣，而不是假设用户想复制这些游戏。

验收标准：

- 每个画像都区分硬性约束、强偏好和软偏好。
- 关键缺失信息是可见的。
- 画像应能在不阅读原始对话的情况下被使用。

### 6.5 机会匹配模块

目的：

找到与开发者画像适配的、有潜力的设计图谱区域。

输入：

- 开发者画像
- 设计知识图谱输出

输出：

- 候选机会区域

职责：

- 将开发者约束和偏好作为评分修正项。
- 尊重硬性约束。
- 避免所有推荐都退化为最安全、最熟悉的模式。
- 提供稳妥型、平衡型和挑战型机会区域。

不得做：

- 生成最终概念。
- 把每个偏好都当作硬过滤条件。
- 隐藏适配度与新颖度之间的取舍。

独立测试用例：

- 给定严格硬性约束，模块应排除不兼容区域。
- 给定强但非硬性的偏好，模块仍可返回带警告的挑战型选项。
- 给定稀疏匹配，模块应解释哪些约束压窄了结果。

验收标准：

- 每个机会区域都包含适配理由和约束取舍说明。
- 在可能的情况下，结果应包含不止一种风险姿态。
- 模块能解释为什么某个有吸引力的区域被拒绝。

### 6.6 机会框架模块

目的：

创建一个把证据连接到潜在概念生成的有边界创意简报。

输入：

- 候选机会区域
- 证据路径
- 开发者画像

输出：

- 机会框架

职责：

- 组装支撑该机会的来源游戏、机制、体验、约束和创新模式。
- 定义推荐变形方式，例如组合、反转、迁移、压缩或替代。
- 基于硬性约束和高风险错配定义禁止方向。
- 让推理足够明确，使人类可以质疑和审查。

不得做：

- 写 polished 的最终概念。
- 引入没有支撑的机制或参考。
- 省略风险理由。

独立测试用例：

- 给定低美术成本、高重玩性的机会，模块应生成带匹配证据路径的框架。
- 给定反对多人联网的硬性约束，框架应禁止依赖多人联网的概念。
- 给定弱证据，框架应将该机会标为低置信度。

验收标准：

- 每个框架都引用来源游戏或有证据支撑的设计论断。
- 每个框架都包含允许变形方式和禁止方向。
- 每个框架都能作为独立创意简报阅读。

### 6.7 概念生成模块

目的：

将机会框架转化为具体、可读的游戏概念。

输入：

- 机会框架

输出：

- 概念卡

职责：

- 在框架允许的设计空间内生成多个概念。
- 让每个概念具体到可以评估。
- 保留参考来源、约束和变形逻辑。
- 避免违反禁止方向的概念。

不得做：

- 发明没有支撑的理由。
- 忽略开发者约束。
- 声称概念一定好玩或成功。

独立测试用例：

- 给定带有明确禁止方向的机会框架，生成的概念应避开这些方向。
- 给定同一个框架，生成的多个概念应有实质差异，而不是同一想法的改写。
- 给定证据较弱的框架，生成的概念应带有适当不确定性。

验收标准：

- 每张概念卡都映射回一个机会框架。
- 每张概念卡都说明参考来源以及与参考来源的差异。
- 每张概念卡都包含制作风险和设计风险。

### 6.8 概念评估模块

目的：

以帮助开发者选择的方式比较生成概念。

输入：

- 概念卡
- 开发者画像
- 机会框架

输出：

- 概念评估
- 排序后的概念集合

职责：

- 评估适配度、可行性、新颖度、风险和证据质量。
- 将概念分类为稳妥型、平衡型或挑战型。
- 解释评分取舍。
- 标出那些有吸引力但与开发者当前能力存在错配的概念。

不得做：

- 把新颖度最高的概念自动视为最好。
- 用数字分数隐藏不确定性。
- 仅因为有风险就移除挑战型概念，除非它违反硬性约束。

独立测试用例：

- 给定一个稳妥但熟悉的概念，以及一个有风险但新颖的概念，模块应区分二者取舍。
- 给定违反硬性约束的概念，模块应拒绝或明确标记。
- 给定证据质量低的概念，即使听起来有吸引力，模块也应降低置信度。

验收标准：

- 每个分数都包含简短解释。
- 每个概念都获得一种定义好的分类。
- 排名帮助比较，但不伪装成客观成功预测。

### 6.9 原型验证模块

目的：

把概念中最大的不确定性转化为最小测试。

输入：

- 概念卡
- 概念评估

输出：

- 原型验证简报

职责：

- 识别最大风险假设。
- 定义最小原型范围。
- 定义可观察的成功和失败信号。
- 明确列出暂时不应该制作的内容。

不得做：

- 使用泛泛的验证建议。
- 将点子扩展为完整制作计划。
- 要求第一轮测试不需要的资产或系统。

独立测试用例：

- 给定一个基于规则操控的概念，模块应测试玩家能否理解并利用规则变化。
- 给定一个低美术约束概念，模块应测试在没有精美美术时反馈是否仍然有效。
- 给定一个声称有重玩性的概念，模块应测试早期变化是否能产生重玩意愿。

验收标准：

- 每份简报都命名一个主要最大风险假设。
- 每份简报都可以在没有完整游戏计划的情况下执行。
- 每份简报都包含具体成功信号和失败信号。

### 6.10 审计与解释模块

目的：

让系统推理过程可以被检查。

输入：

- 游戏设计档案
- 设计论断
- 机会框架
- 概念卡
- 概念评估
- 原型验证简报

输出：

- 解释链路
- 来源引用
- 论断谱系

职责：

- 展示一个概念如何从来源游戏、设计论断和开发者约束推导而来。
- 区分高置信度论断和低置信度论断。
- 让薄弱或推测性推理可见。
- 支持人类对可疑推荐进行抽查。

不得做：

- 重写推荐本身。
- 隐藏缺失证据。
- 将低置信度论断呈现为高置信度事实。

独立测试用例：

- 给定一张概念卡，模块应返回来源机会框架和支撑论断。
- 给定一条可疑论断，模块应显示其质量状态和置信度。
- 给定一条低置信度论断，模块应验证它在解释中被明显降级展示。

验收标准：

- 每个最终推荐都有解释链路。
- 解释链路能识别无支撑或低置信度连接。
- 人类能检查系统为什么推荐某个概念。

## 7. 跨模块规则

### 7.1 先有证据，再生成

概念生成必须位于机会框架下游。系统不得只根据开发者画像直接生成最终概念。

### 7.2 先有证据和置信度，再使用

AI 生成的设计论断可以进入原型图谱，但必须带有证据、置信度和质量状态。低置信度论断必须保持可见，并降低下游推荐置信度。

### 7.3 约束要加权，不要压扁

系统必须区分硬性约束、强偏好和软偏好。硬性约束阻断概念；强偏好和软偏好影响评分和风险警告。

### 7.4 不承诺好玩

系统可以估计适配度、可行性、新颖度和风险。系统不得在原型验证前声称某个概念好玩。

### 7.5 可解释性是产品功能

每个推荐都必须能通过来源游戏、设计论断、开发者约束和变形逻辑进行解释。

## 8. 端到端示例

开发者画像：

- Solo 开发者
- 程序能力强
- 美术能力弱
- 三个月原型目标
- 喜欢系统型策略、短局和重玩性
- 喜欢 Balatro 和 Into the Breach 等游戏
- 不想做长篇叙事内容或在线多人

机会区域：

- 低美术成本
- 短局
- 高重玩性
- 系统深度
- 符号化或棋盘式呈现

机会框架：

- 来源游戏：Balatro、Into the Breach、Baba Is You
- 相关机制：规则修改、紧凑战术空间、熟悉规则认知
- 相关体验：连锁满足感、战术预判、涌现式解题
- 推荐变形：压缩空间，用符号化反馈替代动画，将规则编辑与短局战术 run 结合
- 禁止方向：在线多人、大地图、长脚本剧情、动画密集战斗

概念卡：

- 标题：Ruleforge Tactics
- 一句话概念：一款短局战术游戏，玩家不是直接指挥单位，而是通过编辑小战场的规则来取胜。
- 与参考作品的差异：它借鉴了紧凑战术压力和规则操控，但把每一局包装为一组临时战场法则的构筑。
- 适配理由：它偏向系统而非美术体量，支持短周期原型，并且可以从简单符号化表现开始。
- 主要风险：玩家可能无法足够快地理解规则后果。

原型验证简报：

- 最大风险假设：玩家能否在最初几分钟内理解并享受改变战场规则。
- 最小原型：一个 4x4 棋盘、三种单位、六张规则牌、五场遭遇。
- 成功信号：测试玩家在一局后能解释规则后果，并主动重开尝试不同规则组合。
- 失败信号：玩家认为结果是任意的，或无法预测规则变化会造成什么影响。
- 暂时不要做：战役进度、精美美术、大量内容、局外成长、在线功能。

## 9. 验证策略

系统应在三个层级进行验证。

### 9.1 模块级验证

每个模块都必须基于已知输入产物通过 fixture 测试。生成文本的具体措辞可以变化，但必需字段、禁止行为、证据使用和约束处理必须可测试。

### 9.2 流程级验证

完整流程应使用多个开发者画像进行测试：

- 美术能力弱的 solo 程序员。
- 设计能力强但程序能力弱的团队。
- 追求高新颖度的小团队。
- 对类型有严格厌恶的开发者。
- 时间极其有限的开发者。

每个画像都应产出反映其约束的机会框架、概念卡、评估和验证简报。

### 9.3 人类判断抽查

领域人员可以抽查：

- 游戏档案是否准确。
- 机会框架是否真的有证据支撑。
- 概念是否与参考作品有实质差异。
- 原型简报是否具体到可以行动。

## 10. 成功标准

产品成功，如果：

- 用户能理解每个概念为什么被推荐。
- 概念反映用户约束，但不会过度保守。
- 系统生成具体原型验证简报，而不是泛泛建议。
- 有证据支撑的设计知识能够提升推荐质量。
- 流程能把模糊创意不确定性转化为可比较的项目候选。

产品失败，如果：

- 它表现得像标签搜索引擎。
- 它在没有证据的情况下直接从宽泛提示生成概念。
- 它把薄弱或无支撑的 AI 论断当作高置信度知识。
- 它反复只推荐明显低风险的克隆式点子。
- 它的原型建议过于泛泛，无法指导行动。
