# 技术栈设计 Spec

## 1. 目的

本文档定义独立游戏创意图谱系统的推荐技术栈。它把产品设计转化为明确的前端、后端、数据层和开发环境选择，但不进入具体实现计划。

目标是选择一套支持快速原型、清晰模块边界、可解释图谱推理，以及各产品模块独立测试的技术栈。

## 2. 技术栈决策摘要

系统采用前后端分离架构：

```text
前端：Next.js + React + TypeScript
后端：Python + FastAPI + Pydantic
图数据库：Neo4j
开发运行环境：Docker Compose 管理本地基础设施
LLM 层：模型供应商适配器抽象
进度更新：优先 SSE，轮询作为 fallback
```

这套技术栈让用户界面和工作台体验留在前端，把图谱推理、游戏标注、机会框架、概念生成等核心逻辑留在后端。

## 3. 前端技术栈

### 3.1 核心框架

使用：

- Next.js App Router
- React
- TypeScript

选择理由：

- Next.js 能为工作台型产品提供稳定应用结构。
- React 适合交互式编辑、图谱浏览和概念对比。
- TypeScript 有助于让前端契约与后端产物 schema 对齐。

### 3.2 UI 与样式

使用：

- Tailwind CSS
- shadcn/ui
- Radix UI primitives
- lucide-react icons

选择理由：

- 产品应当像高效工作台，而不是营销页。
- shadcn/ui 和 Radix 能以较低设计系统成本提供可访问控件。
- Tailwind 支持快速迭代，同时让视觉规则保持局部、明确。

### 3.3 数据请求与客户端状态

使用：

- TanStack Query 管理 API 请求、缓存、mutation 状态和刷新行为。
- URL 状态记录筛选条件、选中的图谱节点、选中的概念卡和当前工作区上下文。
- 小型本地组件状态只用于瞬时 UI 交互。

选择理由：

- 系统有大量服务端产物：游戏档案、设计论断、机会框架、概念卡和验证简报。
- TanStack Query 能让服务端状态保持明确、可测试。
- 避免过早引入大型全局客户端状态库，可以降低原型复杂度。

### 3.4 图谱可视化

使用：

- 第一版原型使用 React Flow 做图谱探索体验。
- 只有当图谱规模或布局需求超出 React Flow 能力时，再考虑 Sigma.js。

选择理由：

- MVP 需要的是可理解的图谱检查，而不是大规模图分析。
- React Flow 更容易和工作台界面结合，例如可选节点、侧边详情面板、关系编辑等。

### 3.5 前端职责

前端负责：

- 开发者画像录入与编辑。
- 游戏入库表单。
- 游戏设计档案和设计论断浏览。
- 图谱探索和解释视图。
- 机会框架查看。
- 概念卡对比。
- 原型验证简报展示。
- 长任务进度展示。

前端不负责：

- 图谱遍历逻辑。
- LLM prompt 逻辑。
- 概念评分逻辑。
- 证据路径组装。
- 标准产物的权威校验。

## 4. 后端技术栈

### 4.1 核心框架

使用：

- Python
- FastAPI
- Pydantic v2

选择理由：

- 后端会承载图谱推理、标注抽取、概念生成和产物校验。
- Python 更适合 AI workflow、图数据库集成和数据处理。
- FastAPI 与 Pydantic 能让请求、响应和内部产物契约清晰、易测试。

### 4.2 后端架构风格

后端服务边界应与产品模块保持一致：

- 游戏入库服务
- 游戏设计标注服务
- 设计图谱服务
- 开发者画像服务
- 机会匹配服务
- 机会框架服务
- 概念生成服务
- 概念评估服务
- 原型验证服务
- 审计与解释服务

每个服务都应接收并返回类型化 Pydantic 模型。服务逻辑应避免依赖 HTTP 特定对象，这样可以直接做单元测试。

### 4.3 API 风格

使用：

- REST endpoint 处理产物 CRUD 和工作流动作。
- SSE endpoint 处理长任务进度更新。
- 对不方便使用 SSE 的环境提供轮询 fallback。

选择理由：

- REST 足以支撑 MVP，并且契约容易检查。
- 游戏导入、标注、机会生成、概念生成等操作可能耗时。
- SSE 提供简单的单向进度流，不需要引入完整实时通信层。

### 4.4 LLM 集成

使用：

- 模型供应商适配器抽象。
- 基于类型化产物构造 prompt 输入，尤其是机会框架。
- 生成结果必须通过 Pydantic 结构化校验。

后端不得让生成逻辑直接从宽泛用户文本运行。概念生成必须位于机会框架下游。

选择理由：

- 产品不应绑定某一个模型供应商。
- 类型化输入和输出能让生成失败更早暴露。
- 机会框架能防止系统退化为无约束点子生成器。

## 5. 数据层

### 5.1 主数据库

使用：

- MVP 使用 Neo4j 作为主数据库。

选择理由：

- 核心领域天然是图结构：游戏连接机制、体验、约束、创新模式、来源论断和概念推导路径。
- 推荐质量依赖可遍历的证据路径，而不只是表格记录。
- MVP 使用一个主数据库可以降低复杂度。

### 5.2 PostgreSQL 决策

第一版不使用 PostgreSQL。

只有当产品出现以下需求时再引入 PostgreSQL：

- 多用户账号。
- 权限和组织。
- 独立于图谱的持久任务历史。
- 账单。
- 分析表。
- 不适合放在 Neo4j 中的非图谱业务数据。

### 5.3 数据 Fixtures

早期开发使用 fixture 数据：

- 种子游戏。
- 示例游戏设计档案。
- 示例设计论断。
- 示例开发者画像。
- 示例机会框架。
- 示例概念卡。

选择理由：

- Fixture 数据能让每个模块在完整导入和生成流程稳定前就可测试。
- Fixture 能让前端开发不必等待所有后端路径完成。

## 6. 仓库结构

推荐顶层结构：

```text
backend/
frontend/
data/
  fixtures/
docs/
  superpowers/
    specs/
```

该结构将前端、后端、数据 fixtures 和设计文档清楚分离。

## 7. 测试策略

### 7.1 前端测试

使用：

- 核心工作台组件的组件测试。
- 使用 mock response 的 API 视图集成测试。
- UI 存在后，为主流程加入浏览器 smoke test。

优先测试视图：

- 开发者画像表单。
- 游戏设计档案视图。
- 图谱解释视图。
- 机会框架视图。
- 概念对比视图。

### 7.2 后端测试

使用：

- 每个 service 的单元测试。
- Pydantic 产物 schema 的契约测试。
- Neo4j 图谱遍历集成测试。
- API route 行为测试。

优先测试行为：

- 设计论断保留证据、置信度和质量状态。
- 机会框架引用证据路径。
- 概念生成不能绕过机会框架。
- 低置信度论断会降低下游置信度。
- 硬性约束会阻断不兼容推荐。

### 7.3 端到端测试

使用小型 fixture 流程：

```text
种子游戏
-> 设计论断
-> 图谱
-> 开发者画像
-> 机会框架
-> 概念卡
-> 验证简报
```

第一条端到端测试应使用 fixtures，而不是实时 LLM 调用。

## 8. 本地开发

使用：

- Docker Compose 管理 Neo4j 和未来本地基础设施。
- 本地前端开发服务器。
- 本地后端开发服务器。
- 使用环境变量管理凭据和模型供应商配置。

MVP 应优先保证本地可运行，再考虑托管部署。

## 9. 部署方向

第一版原型优先 local-first。

后续部署可以考虑：

- 前端部署到适合 Next.js 的平台。
- 后端作为 Python 服务部署。
- 根据成本和运维需要选择托管 Neo4j 或自托管 Neo4j。

具体部署细节不属于第一版技术栈决策范围。

## 10. 明确不采用

第一版不使用：

- 纯 Next.js 全栈架构。
- Django 作为主后端。
- 桌面应用壳。
- PostgreSQL 作为第二个主数据库。
- 完整 websocket 实时通信层。
- 大型前端全局状态库。
- 在适配器边界被证明前接入多个 LLM provider 实现。

这些选择可以在原型证明核心流程成立后重新评估。

## 11. 成功标准

技术栈成功，如果：

- 每个产品模块都能独立测试。
- 前端能作为工作台演进，而不吸收后端推理逻辑。
- 后端能用 Pydantic 校验每一种产物结构。
- 图数据库能通过证据路径解释推荐。
- LLM 使用被隔离在类型化适配器边界后。
- 本地开发足够简单，适合快速迭代。

技术栈失败，如果：

- 图谱推理泄漏到前端组件中。
- LLM prompt 分散在 route handler 里。
- 低置信度论断被当成高置信度数据。
- 在 MVP 需要之前引入两个数据库。
- 项目本地运行难度超过原型阶段的合理复杂度。
